        - script: auth/isLoggedIn.lua
      view: pages/ws

Module Metadata

  • Name: Human-readable module identifier
  • Description: Module functionality overview
  • Version: Semantic versioning for compatibility
  • Author/Email: Maintainer contact information
  • License: Legal usage terms (GPL3)

System Integration

  • Dependencies: Required system components
  • BasePath: URL namespace for module routes
  • Database: Associated database name
  • Settings: Module-specific configuration

Route Definition

  • method: HTTP verb (GET, POST, PUT, DELETE)
  • path: URL endpoint relative to BasePath
  • view: Template file to render response
  • preCheck: Middleware scripts execution chain
  • websocket: WebSocket route flag (optional, true/false)

Lua Middleware

  • Script Path: Relative to module's script directory
  • Execution Order: Sequential processing pipeline
  • Context Sharing: Data passed between scripts
  • Early Termination: Scripts can halt request flow

PreCheck Scripts Deep Dive

PreCheck scripts form a powerful middleware chain that executes before the main route handler. Each script in the sequence can modify request context, validate permissions, or terminate the request early.

auth/isLoggedIn.lua

Authentication middleware that verifies user session validity

-- Check if user is authenticated
local session = eocto.getSession("user_id")
if not session then
  eocto.setResponse(401, {error = "Unauthorized"})
  return false
end

parseHeaders.lua

Extracts and processes HTTP headers for downstream use

-- Parse important headers
local userAgent = eocto.getHeader("User-Agent")
local clientIP = eocto.getHeader("X-Real-IP")
eocto.setLocal("client_info", {
  userAgent = userAgent,
  ip = clientIP
})

currentPage.lua

Sets page context and navigation state for the view

-- Set current page context
local path = eocto.getPath()
eocto.setLocal("current_page", path)
eocto.setLocal("page_title", "Home Dashboard")
eocto.setLocal("breadcrumbs", {
  {name = "Home", url = "/OC/"},
  {name = "Dashboard", url = path}
})

session.lua

Manages session data and user preferences

-- Load user session data
local userId = eocto.getSession("user_id")
if userId then
  local userData = eocto.getDataFromCollection(
    "users", {_id = userId}
  )
  eocto.setLocal("user", userData)
end

Technical Implementation Details

Configuration Loading Process

1. YAML Parsing

Configuration file is parsed into Go structs using yaml.v3 library

2. Module Registration

Each module is registered with Fiber router using BasePath as group prefix

3. Route Setup

Individual routes are mapped with their HTTP methods and middleware chains

4. Script Compilation

Lua scripts are pre-compiled and cached for optimal performance

Runtime Execution Flow

Request Interception

Incoming requests are matched against configured routes using Fiber's routing engine

Middleware Chain

PreCheck scripts execute sequentially, each with access to request context and previous results

Context Sharing

Data flows between scripts via Fiber's context locals, enabling complex workflows

Response Generation

Final handler renders view template with accumulated context data

Performance & Optimization

Script Caching

Lua scripts are compiled once and reused across requests, minimizing parsing overhead

Memory Pooling

Lua state objects are pooled and reused to reduce garbage collection pressure

Async Operations

Database and external API calls are handled asynchronously to maintain responsiveness

Configuration Best Practices

Do's ✅

Use descriptive module names and clear BasePath prefixes

Keep PreCheck scripts focused on single responsibilities

Document dependencies and version requirements clearly

Use semantic versioning for module releases

Implement proper error handling in Lua scripts

Don'ts ❌

Avoid overlapping BasePath prefixes between modules

Don't create overly complex PreCheck script chains

Avoid hardcoding sensitive data in configuration files

Don't ignore error handling in middleware scripts

Avoid blocking operations in PreCheck scripts

The Power of Configuration-Driven Architecture

The Octopus framework's YAML-based configuration system with embedded Lua middleware represents a paradigm shift in web application development. By separating routing logic from application code and enabling dynamic script execution, developers can build highly flexible, maintainable, and scalable web applications that adapt to changing business requirements without traditional deployment overhead.
